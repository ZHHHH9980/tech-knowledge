# Agent 行为评测与优化

## 核心原则

执行 `Execute → Evaluate → Optimize → Re-evaluate`。历史轨迹已有 Execute 产物时，从 Evaluate 开始。

把评测与修改分开：先用证据测评并与用户讨论，再获得针对精确修改清单的确认，最后修改和复测。

<HARD-GATE>
在完成 Eval、讨论方案并获得用户对精确修改清单的明确确认前，不得修改任何规则、Skill、测试、业务代码或外部对象。

“搞”“端到端处理”“你看着办”只授权启动本流程，不批准尚未列出的改动。不得把分析请求、旧授权或写权限本身解释为修改批准。
</HARD-GATE>

## 工作流

按顺序维护以下检查项：

1. **建立 Eval 契约**：确认原任务、真实目标、当时可见信息、适用约束、评测范围和成功标准。仅在缺失信息会改变结论时追问。
2. **保存原始证据**：把用户材料作为只读样本。不要先改写成带有主 Agent 结论的摘要再交给 Evaluator。
3. **隔离 Evaluate**：完整读取 [评测量表](references/evaluation-rubric.md)。创建新鲜上下文的 Evaluator 子 Agent，只提供原始材料、当时适用规则和量表，不提供预期答案、疑似问题或拟议修复。给出有限文件清单，禁止扩展搜索，要求单回合返回结构化结果。Evaluator 只读，不得修改任何状态；若迟迟不返回，终止它并按不可用回退处理。
4. **复核 Eval**：检查每条负面结论是否有可定位证据，是否区分事实、推断和未知，是否存在事后偏见或范围外建议。保留分歧，不强行统一。
5. **讨论 Optimize 方案**：提供 2～3 个方案、取舍、潜在副作用、推荐方案和复测方法。优先选择最窄且有效的落点。
6. **展示修改清单**：列出精确文件或对象、具体改动、写入范围、预期行为变化和验证方式，然后停止并等待用户确认。
7. **实施获批改动**：重新读取目标目录规则和适用 Skill，确认 Git 基线，只做清单内修改。原项目权限与安全边界继续生效。
8. **Re-evaluate**：用相同样本、约束、量表和成功标准重放。需要验证 Agent 行为时，使用新的 Executor 生成轨迹，再由新的 Evaluator 评分；需要验证代码时，同时运行相应自动化检查。

如果无法创建隔离 Evaluator，说明自评偏差风险并征得用户同意后再自行评测；不得把自评称为独立 Eval。

## 修复层级

| 问题性质 | 首选落点 |
| --- | --- |
| 可机械检测或阻止 | 测试、lint、校验脚本、业务代码保护 |
| 单项目稳定约束 | 项目 `CLAUDE.md` 或人工维护的 `AGENTS.md` |
| 跨项目判断方法 | 专项 Skill |
| 跨所有任务且长期稳定的红线 | 全局规则 |
| 单次偶发且不可复用 | 只给本次建议，不沉淀 |

规则不能替代可自动化检查。一个案例不足以证明全局规则；先检查反例、复发性、现有规则和触发频率。

## 输出要求

修改前依次输出：

1. 最重要的 1～3 个评测结论
2. 证据化发现、维度评分、未知项和正向行为
3. 优化方案及推荐
4. 待确认修改清单

修改后追加基线与复测对比。没有同量表复测，不得宣称 Agent 行为已经改善。

## 红旗

出现以下想法时停止写入并回到对应阶段：

- “用户说端到端，所以可以直接改。”
- “问题很明显，不需要独立 Eval。”
- “先把规则写了，再让用户看。”
- “多写一条全局规则总不会错。”
- “文件检查通过就等于 Agent 行为改善。”
- “我已经知道答案，可以把结论告诉 Evaluator。”

## 常见误区

| 误区 | 正确处理 |
| --- | --- |
| 只评价最终结果 | 还原当时可见信息和关键决策点 |
| 把隐藏推理当证据 | 只评价可见消息、操作和产物 |
| 只批评不良行为 | 同时记录有效行为，防止修复时破坏它 |
| 用分数代替诊断 | 分数只用于前后对比，修改必须追溯到具体发现 |
| Optimizer 自由发挥 | 每项改动必须对应已确认的 Eval 发现 |
