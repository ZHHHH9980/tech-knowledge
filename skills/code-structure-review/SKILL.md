# Code Structure Review

执行只读的结构审查。除非用户明确要求实现，否则不要修改被审代码。

## 核心目标

把“结构清晰”判断为以下结果，而不是类型、函数或文件数量越多越好：

- 一条关键业务规则只有一个权威决策点。
- 共享状态、配置和生命周期的操作有明确的语义所有者。
- 读路径与写路径共享决策语义，同时保持副作用边界分离。
- 主流程能直接展示业务阶段，边界细节停留在合适的抽象内。
- 每个抽象都降低理解成本或变更放大，不只是搬运参数。

## Review 范围

结合足够的调用者和上下文审查变更行。区分：

- 本次变更新增的问题。
- 本次变更明显加剧的既有问题。
- 与本次变更无关的历史债务。

只报告前两类，不把 Review 扩张为无关的遗留代码清理。

审查重构时同时比较变更前后，不要只看最终代码。重点比较规则实现数量、状态所有者、私有包装类型、调用跳转、可变状态入口和测试契约是否真正收敛。

对纯移动、重命名或机械拆文件按语义结构比较，不能因为代码在 diff 中显示为新增就把历史债务算作本次引入。只有变更增加了规则实现、状态入口、跳转层级或未来修复成本时，才作为新增或加剧的问题报告。若用户明确要求判断某次重构是否达到既定目标，可在总体结论中说明仍未解决的结构缺口，但不要把未变严重的旧问题伪装成回归 finding。

## Review 标准

### 1. 明确职责与状态的语义所有者

先判断“谁拥有这段业务状态、约束和生命周期”。

当一组操作共享可变状态、配置、不变量和固定执行顺序时，优先让一个内聚的领域对象、receiver、模块或组件承载它们。把稳定共享的依赖放在所有者上，把每次操作独有的事实保留为显式参数。

重点检查：

- 多个自由函数反复传递同一份 `state`、`settings`、事务或上下文。
- 同一状态机的推进步骤散落在多个没有独立语义的 helper 中。
- 主流程虽然短，但读者必须跨多个包装类型追踪同一份状态。
- 任意函数都能修改状态，却没有单一入口维护生命周期和顺序。

不要机械地把自由函数改成 method。无状态纯函数可以保持独立；只有共享状态、规则或生命周期确实构成稳定边界时，才建议引入语义所有者。也不要把独立生命周期和无关副作用塞进一个“大管家”对象。

### 2. 收敛关键业务规则的唯一真相源

对每项状态迁移、阈值、时间窗口、资格、计费、权限、幂等或兼容规则，追踪所有消费者，包括：

- 写入、落库和定时结算路径。
- 查询、展示和只读预演路径。
- 同步、异步、重试和恢复路径。
- 缓存、事件、通知和外部调用路径。

如果多个路径表达同一张决策表，要求它们共享一个权威的纯规则函数、规则对象或显式转换表；让各路径继续分别承担查询、持久化和通知等副作用。

使用“变更放大测试”验证结构：假设修改一个阈值、状态集合或时间边界，需要同步修改多少处分支？同一业务决策需要改多处时，通常说明规则尚未收敛。

不要因为代码长得相似就强行复用。只有业务语义、不变量和未来变更原因相同，才能共享规则。读路径和写路径可以共享状态决策，但不要为了复用把只读投影与状态写入混成一个有隐藏副作用的函数。

### 3. 检查抽象是否有净价值

检查本次新增或修改的私有 type、struct、helper、wrapper、hook 和 component。每个抽象至少应明确提供一项价值：

- 承载独立不变量或领域概念。
- 建立状态或副作用的所有权边界。
- 被多个真实调用点复用。
- 隔离外部依赖、兼容策略或复杂边界。
- 减少业务规则的修改点或调用方认知负担。

将以下情况视为收敛信号，而不是继续拆分的理由：

- 只在一个调用点构造、只为向下透传参数的包装 struct。
- 仅转调下一个函数、没有新增策略或边界的 helper。
- 围绕同一份可变状态创建多个一次性参数载体。
- 提取后主流程变短，但真实业务顺序被分散到更多文件和跳转中。

单调用点本身不是问题。即使只调用一次，只要抽象承载了独立不变量、策略边界、可测试契约或复杂隔离，它仍然有价值。Review 必须说明抽象缺失的具体价值，不能只用调用次数下结论。

### 4. 让核心流程保持为编排概要

核心入口应直接展示业务阶段，不嵌入详细条件、持久化细节、重试、清理、格式转换或兼容回退。

优先形成类似结构：

```go
func HandleRequest(ctx context.Context, request Request) error {
    input, err := validateRequest(ctx, request)
    if err != nil {
        return err
    }

    result, err := executeTransition(ctx, input)
    if err != nil {
        return err
    }

    return publishResult(ctx, result)
}
```

将输入归一化、空状态、兼容分支、重试回退、部分失败恢复、幂等去重、资源释放和外部错误翻译留在聚焦的边界抽象中。

同时避免“虚假的短主流程”：如果每一步只是跳进另一个一次性 wrapper，且读者仍无法看到状态所有者和规则顺序，应按抽象碎片化报告。

### 5. 把体量当作信号，不当作结论

优先遵循目标项目已有的文件和函数体量规则。项目没有明确标准时，把手写源码接近或超过 400 行、函数接近或超过 50 行视为深入检查信号。

只有确认文件或函数混合了独立职责后才报告问题，并给出按领域或职责划分的具体边界。不要机械标记生成代码、锁文件、vendored 依赖、机器生成协议绑定，以及结构上必须保持单文件的数据 fixture。

大型手写 API schema、协议、配置和测试文件仍需检查；存在清晰领域边界时才建议拆分。

### 6. 在最窄的语义所有者处说明关键意图

用注释解释无法仅从操作推导的生命周期、业务原因、不变量和正确性顺序，重点包括：

- 状态机、状态迁移、阈值、时间窗口和跨日规则。
- 计费、余额、权益、配额、资格和权限判定。
- 幂等、去重、锁、fencing、CAS 和并发竞态裁决。
- 持久化、缓存、事件、通知和外部调用之间的执行顺序。
- 重试、回退、fail-open / fail-closed、补偿和部分失败恢复。
- 协议版本、历史数据、兼容分支和迁移过渡策略。
- 表面为读取但会刷新、上报、持久化或产生其他隐藏写入的路径。

优先把说明放在规则函数、状态所有者或决策簇入口。若短编排已通过领域方法名清晰表达步骤，并且顺序与不变量已在该语义所有者处说明，不要要求调用点重复添加逐步注释。

只有调用点引入了额外的业务原因、跨抽象顺序或副作用约束时，才要求就地注释。

有效注释：

```go
// 先持久化终态再发布事件，确保重试保持幂等。
if err := finalizeSession(ctx, session); err != nil {
    return err
}
```

无效注释：

```go
// 结束会话。
if err := finalizeSession(ctx, session); err != nil {
    return err
}
```

### 7. 按语义选择参数结构

把三个以上同类依赖或六个以上位置参数视为检查信号，但不要机械创建参数 struct。

按以下语义选择：

- 对公开 use case 输入或稳定的数据契约使用具名 params struct。
- 对多步操作共享的状态和配置使用内聚的所有者或 receiver。
- 对单次操作独有的少量事实使用显式参数。
- 对可选配置和默认值较多的公共 API，按项目惯例考虑 options。

避免只为满足参数数量规则而创建单调用点包装。推荐时说明新结构表达的领域含义、所有权或不变量。

### 8. 用测试守住行为不变的重构

审查声称“行为不变”的重构时，确认原有状态边界、时间边界、错误路径、幂等和只读不写回等契约仍被测试覆盖。测试应迁移到新的内聚 API，而不是继续绕过语义所有者调用已废弃的内部碎片。

没有运行测试时明确标记未知，不要仅凭代码移动宣称行为等价。不要为了结构指标要求删除有价值的边界测试。

## 收敛判断方法

先用领域语言描述重复的业务决策，再选择最小的权威落点，不预设必须使用 class、method、receiver、规则对象或某种命名：

1. 写出规则的输入、输出、不变量和所有消费者。
2. 判断消费者共享的是同一业务决策，还是只有代码形状相似。
3. 把真正共享的决策收敛到一个无副作用的权威落点。
4. 让查询、展示、落库、通知等消费者分别保留自己的副作用边界。
5. 只有多步操作确实共享状态和生命周期时才建立状态所有者；纯计算保持普通纯函数即可。

最终以变更放大、隐藏写入、状态入口和认知跳转是否减少来判断效果，不以是否采用某种代码拓扑判断。

## Review 流程

1. 锁定精确的 PR、MR、commit 或 diff，以及目标项目的结构规则。
2. 列出变更文件并测量 reviewed revision 的行数，排除生成和 vendored 文件。
3. 找出核心入口、可变状态、外部副作用和关键业务规则。
4. 为每项关键规则建立简短映射：规则 → 权威所有者 → 读写消费者 → 副作用入口。
5. 比较各路径是否重复实现同一决策，执行一次变更放大测试。
6. 检查变更涉及的私有抽象是否承载独立语义，识别一次性参数包装和透传 helper。
7. 检查核心流程是否是清晰的业务概要，而非隐藏跳转或内嵌边界细节。
8. 检查关键意图是否位于最窄语义所有者，避免同时漏注释和重复注释。
9. 对行为不变的重构检查测试契约；未执行测试时记录未知。
10. 只输出有具体代码证据、由本次变更新增或加剧的可执行发现。

## Finding 要求

每条 finding 必须包含：

- 优先级：`P1` 表示可能造成规则分叉、状态错误或需要阻断合并的问题；`P2` 表示重要的可维护性问题。
- 1～10 的置信度。
- 精确文件和行号。
- 具体代码证据及受影响的调用路径。
- 当前结构如何增加规则分叉、状态所有权或理解成本。
- 可执行的收敛建议，明确权威规则、语义所有者、保留的副作用边界或应删除的低价值包装。

使用格式：

```markdown
1. `[P1][10/10]` 简洁的 finding 标题
   `path/to/file.go:123`

   证据与影响。

   明确的规则归属、抽象边界或收敛建议。
```

把同一根因的多个症状合并为一条 finding。不要只说“建议重构”“抽个 helper”或“补注释”。如果变更已经满足标准，明确说明没有结构 finding，不要为了填满模板制造问题。

## 输出要求

结论优先，并明确说明：

- 变更是否满足结构标准。
- 是否存在同一业务规则的多个实现及其权威所有者。
- 是否存在低价值包装、职责碎片化或不清晰的状态所有权。
- 哪些核心流程仍嵌入边界细节或隐藏副作用。
- 哪些关键意图无法从语义所有者附近获得，而不是机械统计逐步注释。
- 哪些文件违反项目体量规则或跨过默认检查信号。
- 行为不变是否经过测试验证，哪些内容仍属未知。
- 是否排除了生成文件，以及是否修改了任何代码。

不要报告没有维护性影响的风格偏好。聚焦最窄且真正有效的收敛点，不建议无证据的广泛重写。
