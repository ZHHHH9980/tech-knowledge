# 技术方案数据访问评审

基于实际接口、调用路径、SQL、DDL、索引和容量证据给出可执行的评审结论。不要只复述 Tech Design，也不要把个人偏好包装成 Finding。

## 评审边界

- 默认只读评审；除非用户明确要求，否则不要修改代码、文档、数据库或远端状态。
- 优先检查用户给出的 PRD、Tech Design、MR/PR 和代码；缺少材料时再搜索关联实现。
- 区分设计、实现和生产表现：
  - 文档描述意图。
  - 代码和 SQL 决定实际行为。
  - `EXPLAIN`、监控和压测决定容量结论。
- 缺少数据量、QPS、执行计划或监控时，将容量结论标记为“未验证”，不要猜测能扛住。

## 评审流程

### 1. 建立证据链

尽可能读取：

1. PRD 和业务规则。
2. Tech Design、RFC 或 ADR。
3. Proto、OpenAPI、GraphQL schema 或接口定义。
4. Handler、service、logic、repository、DAO 等调用路径。
5. 实际 SQL、ORM 查询和批处理逻辑。
6. Migration、DDL、约束、分区和索引。
7. 相邻模块或历史接口中的分页实现。
8. `EXPLAIN (ANALYZE, BUFFERS)`、`pg_stat_statements`、监控或压测结果。

引用具体证据。不要仅凭字段名推断业务关系，也不要仅凭索引名称推断查询能够命中。

### 2. 检查接口查询边界

逐个检查所有返回集合或批量处理数据的接口：

- list、search、history、feed；
- sync、poll、增量同步；
- follower/following；
- 定时任务、队列消费者和后台批处理。

为每个接口回答：

| 检查项 | 必须回答的问题 |
|---|---|
| 查询范围 | 全量查询还是有界查询？ |
| 单次上限 | 服务端是否有 `limit`，最大值是多少？ |
| 分页方式 | keyset/cursor、offset，还是没有分页？ |
| 排序规则 | 是否稳定，是否包含唯一 tie-breaker？ |
| 游标字段 | 是否覆盖完整排序字段？ |
| 数据变化 | 翻页期间是否会重复、遗漏或死循环？ |
| 响应协议 | 是否返回 `next_cursor`、`has_more`？ |
| 实际 SQL | 是否真的有相应的 `WHERE`、`ORDER BY`、`LIMIT`？ |
| 索引 | 是否匹配过滤条件和排序？ |
| 参考实现 | 项目中是否已有成熟分页可以复用？ |

#### 全量查询

只在存在严格、可证明的小规模上限时接受全量查询，例如配置枚举或固定条目。要求方案解释为什么必须全量，并评估：

- 最大记录数和增长方式；
- DB 扫描、排序与聚合；
- 响应体、序列化和应用内存；
- QPS 增长后相同成本如何被放大。

没有严格上限的高频全量查询应形成 Finding。

#### 分页查询

优先使用 keyset/cursor 分页。检查游标是否覆盖完整排序：

```sql
ORDER BY created_at DESC, id DESC
```

游标至少包含 `created_at + id`，查询应类似：

```sql
WHERE (created_at, id) < (?, ?)
ORDER BY created_at DESC, id DESC
LIMIT ?
```

若排序包含状态优先级，游标与索引也必须包含该状态表达式。不要仅使用非唯一时间字段作为游标。sync/poll 同样必须有服务端批次上限。

#### 复用已有实现

搜索并比较项目中的：

- `before_id`、`after_id`；
- `next_cursor`、`has_more`；
- `page_size`、`LIMIT`；
- keyset tuple comparison。

比较 API 命名、默认和最大 page size、cursor 编码、排序 tie-breaker、客户端去重语义、SQL 和索引形状。已有成熟实现时优先复用；不能复用时要求方案说明差异。

### 3. 检查数据库关系

先根据业务生命周期画出逻辑关系，再检查约束：

```text
parent
 ├── child_a
 └── child_b
```

对每个关联回答：

| 子表 | 关联字段 | 父表 | 是否同库同域 | 能否独立存在 | 当前 FK | 查询索引 | 建议 |
|---|---|---|---|---|---|---|---|

满足以下条件时优先考虑 FK：

- 父子表位于同一数据库和业务域；
- 子记录没有父记录就没有业务意义；
- 子记录不允许先于父记录创建；
- 父记录删除后不允许保留孤儿数据；
- 写入、迁移和删除成本可以接受。

以下情况不要机械增加 FK：

- 跨服务、跨数据库或跨 ownership；
- 数据需要独立归档或保留；
- 子记录允许异步先写入；
- 分库分片或离线同步链路；
- 删除和保留策略尚未确定。

FK 建议必须说明 `RESTRICT/NO ACTION`、`CASCADE` 或 `SET NULL`。默认不要擅自选择 `CASCADE`，大表级联删除可能形成超大事务。

使用 orphan query 验证现状：

```sql
SELECT child.parent_id
FROM child
LEFT JOIN parent ON parent.id = child.parent_id
WHERE parent.id IS NULL
LIMIT 100;
```

明确说明：FK 保证数据完整性，不负责提升普通查询性能。

### 4. 从实际 SQL 推导索引

检查实际 SQL 的：

- `WHERE`、`JOIN`；
- `ORDER BY`、`GROUP BY`、`DISTINCT`；
- `LIMIT`；
- partial predicate；
- 定时任务扫描条件。

例如：

```sql
WHERE conversation_id = ?
ORDER BY sequence_no DESC
LIMIT 20
```

对应索引通常应从 `(conversation_id, sequence_no DESC)` 评估。

检查规则：

- 等值过滤字段通常位于联合索引前部；
- 范围和排序字段位于等值字段之后；
- 检查最左前缀、排序方向和 tie-breaker；
- 稀疏状态优先评估 partial index；
- 需要减少 heap fetch 时评估 covering index；
- 检查新索引是否被已有索引覆盖；
- 评估写放大、磁盘和 autovacuum 成本；
- 分区表检查索引是否覆盖全部分区；
- `CREATE INDEX CONCURRENTLY` 必须与迁移事务模式兼容。

不要把“索引包含这些字段”等同于“索引匹配这条查询”。

### 5. 评估容量和慢 SQL 风险

分别评估：

```text
请求量增长 = 当前 QPS × 增长倍数
数据量增长 = 当前增长速度 × 时间 × 增长倍数
```

重点检查：

- 每个请求执行多少条 SQL；
- 是否重复执行 `COUNT`、`SUM` 等聚合；
- 是否存在无界扫描、排序、去重或聚合；
- 是否有高频后台扫描；
- 是否存在 `FOR UPDATE`、热点行或长事务；
- 是否存在固定 batch、固定 worker 或 backlog；
- 分区只是裁剪，还是确实提供水平扩容；
- 数据归档与保留策略；
- WAL、连接池、I/O、锁和 autovacuum 压力。

注意：QPS 增长 100 倍和数据量增长 100 倍是两个不同维度，可能同时发生。

没有生产基线、目标数据量执行计划、压测、DB 规格和 backlog 指标时，不要声称“能扛住”。

### 6. 检查推送通知的账号与设备边界

凡是涉及推送（TIM Push、APNs、FCM、厂商通道等）的方案，先确认一个基本事实：**业务数据是 By 账号的，推送 token 是 By 设备的**。App 端经常多账号（同设备切换账号、同账号多设备），两者不是一一对应，任何隐含“一个用户 = 一个设备”的设计都要形成 Finding。

逐项回答：

| 检查项 | 必须回答的问题 |
|---|---|
| token 归属 | device token 表的主键是设备还是账号？切换账号时 token 如何转移或解绑？ |
| 绑定时机 | 登录、登出、切换账号时是否重新绑定/解绑 token？失败重试吗？ |
| 隔离性 | 账号 A 登出后，账号 A 的消息是否还会推到该设备（推给了正在登录账号 B 的人）？ |
| 多设备 | 同一账号多设备登录时，推一台还是全推？已读状态是否跨设备同步？ |
| 接口隔离 | 拉取通知列表、未读数、已读上报等接口是按账号鉴权，还是隐含用了设备维度的标识？ |
| 残留数据 | 设备卸载重装、token 失效后，脏 token 如何清理？是否会持续空推？ |
| 敏感内容 | 推送 payload 是否包含账号敏感内容？错推到他人设备时泄露什么？ |

典型 P0：登出/切号后未解绑 token，导致前账号的私信、订单等敏感通知推送到已易主的设备。这类问题既是功能 bug 也是隐私事故，必须阻断。

验证方式：给出切号场景的集成测试或手工验证步骤（设备登录 A → 登出 → 登录 B → 触发 A 的通知，确认设备不收到），以及 token 绑定表的 orphan/残留查询。

### 7. 检查订阅扇出与频控

凡是“某事件触发 → 通知一批订阅者”或“用户订阅多个源 → 每个源都产生消息”的设计，必须评估扇出乘积和频控，防止消息爆炸。

先写出扇出公式，用具体数字算一遍：

```text
单事件消息量 = 订阅者数 × 每次事件的消息条数
单用户收到量 = 关注源数量 × 每源事件频率 × 每次事件的消息条数
```

例如：一个用户订阅一个角色能收到 1～3 条消息，关注 200 个角色、每个角色每天更新一次，则该用户每天收到 200～600 条推送——这就是必须频控的证据。

逐项回答：

| 检查项 | 必须回答的问题 |
|---|---|
| 扇出上限 | 单事件最大订阅者数是多少？头部源（大 V、热门角色）的扇出量级？ |
| 用户侧频控 | 单用户单位时间收到的推送有无上限？超限后是丢弃、合并还是延迟？ |
| 事件侧频控 | 单个源单位时间能触发几次推送？连续触发是否去重/合并？ |
| 合并策略 | 多条同类消息能否折叠为一条摘要（“你关注的 N 个角色有更新”）？ |
| 静默与偏好 | 是否尊重免打扰时段、用户级/源级的推送开关？ |
| 扇出实现 | 扇出是同步循环写还是走队列？头部源事件是否会打爆下游（DB 写入、推送通道 QPS）？ |
| 削峰 | 队列消费速率、批量大小、backlog 告警是否定义？ |
| 幂等 | 重试或重复消费时，用户会不会收到重复推送？ |

典型 P0：无任何用户侧频控且扇出乘积无上限——头部用户被消息轰炸导致关闭推送或卸载，同时推送通道和 DB 承受无界写放大。

频控方案要求明确：限流的 key（用户/源/用户×源）、窗口、阈值、超限行为（丢弃/合并/降级为站内信），以及这些计数器自身的存储与容量（Redis 结构、过期策略）。

验证方式：用头部源的真实订阅数压测扇出链路；构造高频事件验证频控生效；监控单用户日推送量分布（P50/P99）。

## Finding 契约

只报告真实、具体、可执行的问题。正常设计放在覆盖表；证据不足的风险放在“待确认问题”。

### 优先级

- **P0**：必须阻断合并。包括明显 bug、关键业务语义不明、单一函数承担过多独立职责，或状态管理混乱且缺少内聚的语义所有者。数据访问场景还包括无严格上限的高频全量查询、会稳定遗漏数据的分页、关键热路径必然退化的 SQL，或能够稳定产生错误数据的关系设计。推送场景还包括切号后通知错推到他人设备的账号隔离缺陷，以及扇出乘积无上限且无频控的消息爆炸设计。
- **P1**：可能造成规则分叉、状态错误、显著容量风险，或明显增加后续变更风险的重要问题。
- **P2**：重要的可维护性、可观测性或数据库治理问题。

### 置信度

每条 Finding 标记 `1～10`：

- `9～10`：接口、调用路径、SQL、DDL 和索引提供直接证据；
- `7～8`：代码证据充分，但缺少生产数据量、执行计划或监控；
- `5～6`：风险合理，但关键事实尚未确认；
- 低于 `5`：不要作为 Finding，放入“待确认问题”。

优先级表示影响，置信度表示证据强度，不要混用。

### 每条 Finding 必须包含

1. 优先级和 `1～10` 置信度。
2. 精确文件和行号。
3. 具体代码、SQL、DDL 或接口证据。
4. 受影响的完整调用路径。
5. 当前结构如何增加规则分叉、状态所有权、无界成本、数据完整性风险或理解成本。
6. 可执行的收敛建议，明确：
   - 权威分页规则；
   - 语义和状态所有者；
   - FK 与删除策略；
   - 索引对应的实际 SQL；
   - 保留的副作用边界；
   - 应删除的重复或低价值包装。
7. 验证修改有效的方法。

代码或 DDL 存在时必须给出 `path:line`，不要只引用文档。只有设计文档、尚无代码时，使用文档 URL、精确章节和原始规则，并明确“实现尚未验证”。

### Finding 格式

严格使用：

```markdown
1. `[P0][10/10]` 简洁的 Finding 标题
   `path/to/file.go:123`

   证据：具体代码、SQL、DDL 或接口行为。

   调用路径：

   `HTTP/gRPC API → service/logic → repository/data → SQL → table/index`

   影响：说明触发条件、受影响规则，以及当前结构如何增加规则分叉、状态所有权、数据完整性、性能或理解成本。

   收敛建议：明确权威规则、语义所有者、FK/索引设计、保留的副作用边界，以及应删除的重复实现。

   验证：给出测试、执行计划、orphan 检查、压测或监控方式。
```

## 输出格式

### 结论

只使用以下结论之一，并给出一句理由：

- 可以接受；
- 有条件接受；
- 必须修改后再评审；
- 证据不足。

### Findings

按 P0、P1、P2 排序，再按置信度排序。使用 Finding 契约，不要把建议、优点或猜测混入 Findings。

### 接口覆盖

| 接口 | 全量/分页 | 单次上限 | 游标与排序 | SQL/索引 | 参考实现 | 结论 |
|---|---|---:|---|---|---|---|

### 数据库关系

| 子表 | 父表 | 关联字段 | 能否独立存在 | 当前 FK | 查询索引 | 结论 |
|---|---|---|---|---|---|---|

### 推送与频控覆盖

涉及推送的方案必须填写；不涉及时写明“本方案不涉及推送”。

| 场景 | token 归属与切号处理 | 扇出乘积（含头部估算） | 用户侧频控 | 事件侧频控/合并 | 结论 |
|---|---|---|---|---|---|

### 容量结论

明确区分：

- 已确认安全；
- 静态分析合理但未压测；
- 已发现扩容阻塞点；
- 缺少指标，无法判断。

### 待确认问题

对不足以形成 Finding 的风险列出：

1. 缺少什么事实；
2. 为什么影响结论；
3. 应读取什么代码、指标或由谁确认。

### 最小行动清单

按优先级给出直接对应 Findings 的最小修改和验证步骤。不要顺手重构无关代码。

如果没有达到 P0～P2 的问题，明确写：

```text
未发现需要阻断或要求修改的问题。
```

同时列出已检查的接口、SQL、表关系和索引，不要为了显得评审充分而制造 Finding。
