# Go Code Readability Review

执行只读审核。除非用户明确要求修改，否则不要编辑被审代码。

## 审核边界

只报告本次变更新增或明显加剧的问题。阅读足够的调用者、类型定义和测试来理解变更，但不要把审核扩张为历史代码清理。

本层只回答“代码是否容易被准确理解和安全修改”。以下内容只记录为待进入其他审核层的风险，不在没有证据时下结论：

- 业务逻辑是否正确。
- 是否存在数据竞争、死锁或内存模型问题。
- 性能、容量和资源消耗是否满足目标。
- 安全、权限和输入信任边界是否可靠。

如果可读性问题直接掩盖了明显 bug，应报告该 bug，并明确它需要正确性层确认，不要为了维持层级边界而隐瞒证据。

## 审核标准

### 1. 用类型表达真实数据形状

要求公开输入、领域对象和稳定协议使用能表达约束的具体类型。

重点检查：

- 已知字段集合的 JSON object 是否仍用 `map[string]any`。
- 请求 DTO 是否使用裸 `interface{}` 或 `any` 隐藏顶层数据形状。
- 调用方是否反复执行类型断言、字符串 key 查找和零散校验。
- 多个同类位置参数是否缺少稳定的具名输入类型。
- 一个通用结构是否承载多个互斥状态，却没有判别字段或清晰不变量。

已知协议优先使用具体 struct：

```go
type ImagePayload struct {
	URL    string `json:"url"`
	Width  int32  `json:"width"`
	Height int32  `json:"height"`
}
```

仅在真正无模式的系统边界使用 `map[string]any`，并尽快校验和转换为领域类型。不要机械禁止所有 `any`：日志字段、通用容器和确实需要保留任意 JSON value 的边界可以合理使用。

把框架解析兼容性与可读性分开：裸 `any` 让契约不可见属于本层问题；某个解析器是否会运行时报错属于正确性层，需要通过真实入口测试确认。

### 2. 使用一致的领域命名

检查名称是否让读者无需展开实现即可判断：

- 数据是什么，而不是数据的数据类型。
- 函数执行动作、查询状态还是产生副作用。
- 布尔值的真假分别表示什么。
- 缩写是否符合 Go 与目标项目惯例。
- 同一业务概念是否在不同层使用不同名称。

不要报告纯个人偏好的同义词替换。只有命名造成歧义、掩盖副作用、混淆生命周期或迫使读者频繁跳转时才形成 finding。

### 3. 让控制流直接展示主路径

优先让函数按业务阶段顺序展开，并使用早返回处理前置条件和错误。

重点检查：

- 多层嵌套是否遮蔽 happy path。
- 同一条件是否在函数不同位置重复判断。
- `switch`、布尔组合或否定条件是否让状态空间难以理解。
- 错误处理是否与产生错误的操作相隔太远。
- `defer`、锁、事务、goroutine 和资源释放的生命周期是否能从局部读懂。
- 核心流程是否被多个只做转发的一次性 helper 打散。

不要为了缩短函数机械提取 helper。只有 helper 提供清晰语义、隔离边界细节或被真实复用时，才降低理解成本。

### 4. 让状态和副作用可见

检查读者能否从函数签名、receiver 和调用顺序判断：

- 谁拥有可变状态。
- 哪一步会写数据库、缓存、文件或外部服务。
- 哪些调用只读，哪些调用可能刷新、上报或触发异步任务。
- 锁、事务和取消信号覆盖的代码范围。
- 输入对象是否会被修改或跨 goroutine 共享。

如果共享状态和生命周期稳定，优先由 receiver 或内聚组件承载；如果只是无状态计算，保持纯函数。不要机械地把所有自由函数改成 method。

### 5. 要求抽象提供净价值

检查新增的 interface、泛型、wrapper、helper 和 params struct 是否至少提供一项价值：

- 表达稳定领域概念或不变量。
- 隔离外部依赖或副作用。
- 收敛多个真实调用点。
- 减少调用方需要理解的细节。
- 建立可测试的契约边界。

把以下情况视为风险信号：

- 只有一个实现和一个消费者，却没有替换、隔离或测试价值的 interface。
- 只转发参数、错误和返回值的 helper。
- 只为减少参数数量而创建、没有领域含义的一次性 params struct。
- 需要跨多个小文件才能拼出一个简单顺序流程。

单实现、单调用点和较长函数都不是独立 finding。必须说明它们具体增加了哪些认知跳转、歧义或修改成本。

### 6. 用注释解释原因和不变量

要求注释解释代码本身无法表达的内容：

- 为什么必须按当前顺序执行。
- 某个零值、阈值或特殊状态代表什么。
- 锁、幂等、重试和补偿保护什么不变量。
- 为什么采用兼容、降级或临时实现。

不要要求重复代码表面动作的注释。导出标识符缺少注释只有在目标项目或 lint 明确要求时才报告。

### 7. 把体量当作深入检查信号

优先遵循项目规则。项目没有规则时，可把手写文件接近 400 行、函数接近 50 行作为深入阅读信号，但不能仅按行数报告。

只有代码混合多个独立职责、状态生命周期或变更原因时，才建议拆分，并明确拆分边界。排除生成代码、vendored 依赖、协议绑定、锁文件和机械 fixture。

## 审核流程

1. 锁定精确 diff、基线和目标项目规则。
2. 列出手写 Go 文件，排除生成代码与依赖。
3. 找到入口、核心类型、状态所有者和外部副作用。
4. 沿主要调用路径阅读足够上下文，不只看变更行。
5. 按类型表达、命名、控制流、状态可见性、抽象价值、注释和体量逐项检查。
6. 区分可读性 finding 与需要其他层确认的正确性、并发、性能或安全风险。
7. 合并同一根因的多个症状，只输出有具体证据且可执行的 finding。

## Finding 要求

每条 finding 必须包含：

- 优先级：`P1` 表示显著遮蔽业务意图、容易导致错误修改；`P2` 表示明确增加理解或维护成本。
- 1～10 的置信度。
- 精确文件和行号。
- 具体代码证据和受影响的阅读路径。
- 为什么现有表达容易被误解。
- 最小的改进方向，不要求无关重写。

使用格式：

```markdown
1. `[P1][9/10]` 请求类型没有表达稳定的 payload 结构
   `path/to/file.go:123`

   证据与理解成本。

   建议使用的类型边界或控制流调整。
```

不要报告：

- `gofmt` 已能自动解决的格式差异。
- 没有维护影响的个人风格偏好。
- 仅凭函数、参数或文件行数得出的结论。
- 与本次变更无关的历史债务。
- 没有阅读调用者就推断出的抽象问题。

## 输出要求

先给出是否通过 Go 可读性审核的结论，再列 findings。最后简短说明：

- 已审核的 diff 或 revision。
- 排除的生成文件。
- 需要进入 correctness、concurrency、performance 或 security 层确认的风险。
- 未读取或未验证的内容。
- 是否修改了代码。

没有 finding 时明确说明通过，不要为了填充结果制造问题。
